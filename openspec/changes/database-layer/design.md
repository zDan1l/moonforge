## Context

MoonForge is a new application with no existing database schema. The current Prisma schema file exists but contains only boilerplate - no models are defined. This is the foundational layer that all other features depend on: user authentication, project storage, file versioning, and chat history all require persistent storage.

**Current State:**
- Prisma schema exists at `apps/api/prisma/schema.prisma` but has no models defined
- Prisma config (`prisma.config.ts`) reads DATABASE_URL from `../../.env`
- PostgreSQL running via `docker-compose.dev.yaml` on port `5445:5432`
- Database: `postgres`, User: `postgres`, Password: `postgres`
- Prisma dependencies already installed (`@prisma/client`, `@prisma/adapter-pg`, `prisma`)
- No `apps/api/src/lib/` directory exists yet
- No migrations folder exists yet

**Constraints:**
- PostgreSQL is the chosen database (already in docker-compose)
- Prisma ORM with client output to `apps/api/src/generated/prisma`
- Must support MoonForge's two-phase workflow: Setup (generate) and Refine (versioning)
- Type safety is critical - all future TypeScript types derive from this schema
- Use `pnpm` for package management (monorepo)

## Goals / Non-Goals

**Goals:**
- Define complete database schema with all 5 models (users, projects, project_versions, project_files, chat_messages)
- Establish proper relationships with cascade deletes where appropriate
- Add indexes for performance on commonly queried fields
- Configure enums for status fields (projects.status, project_files.file_source, chat_messages.role)
- Set up Prisma client singleton pattern in `apps/api/src/lib/prisma.ts`

**Non-Goals:**
- Database migrations (will be created separately)
- Seed data (will be added later if needed)
- Query optimization beyond basic indexes
- Backup/replication strategy (handled at infrastructure level)

## Decisions

### 1. UUID vs Integer Primary Keys

**Decision:** Use `UUID` for all primary keys.

**Rationale:**
- Non-sequential IDs prevent enumeration attacks
- Distributed systems friendly (can generate IDs without coordination)
- Prisma's built-in `@default(uuid())` works well with PostgreSQL
- URLs with UUIDs are more user-friendly than long integers

**Alternatives considered:**
- Auto-increment integers: Simpler but predictable
- CUIDs: Prisma-native but UUIDs are more standard

### 2. Cascade Delete Behavior

**Decision:** Use cascade deletes for child entities.

**Rationale:**
- When a user is deleted, all their projects should be removed
- When a project is deleted, all versions, files, and chat messages should be removed
- When a version is deleted, all associated files and messages should be removed
- Simplifies cleanup logic in application code

**Implementation:**
```prisma
relationMode = "prisma" // Default
// onDelete: Cascade on all relation fields
```

### 3. File Storage Strategy

**Decision:** Store file contents as TEXT in database (not blob storage).

**Rationale:**
- Project files are primarily code (text-based)
- Enables querying file contents if needed
- Simpler architecture for MVP - no additional storage service
- PostgreSQL TEXT can handle large files (up to 1GB)

**Trade-off:** For very large binary files, this won't scale. Future enhancement could move large files to S3 with database storing references.

### 4. Version Number Implementation

**Decision:** Store version_number as INTEGER, auto-incremented per project.

**Rationale:**
- Simple sequential versioning (1, 2, 3...)
- Easy to query "latest version" with `max(version_number)`
- No need for complex semantic versioning in MVP

### 5. file_source Enum Values

**Decision:** Three values: `template`, `ai_generated`, `modified`.

**Rationale:**
- Matches PRD requirements exactly
- `template`: Files from the template base (never change)
- `ai_generated`: Files generated by Claude, not yet modified
- `modified`: Files that have been changed via refine workflow

## Database Schema

### users

```prisma
model users {
  id        UUID   @id @default(uuid())
  email     String @unique
  name      String?
  created_at DateTime @default(now())
  updated_at DateTime @updatedAt

  projects  projects[]
}
```

### projects

```prisma
model projects {
  id          UUID   @id @default(uuid())
  user_id     UUID
  name        String
  description String  @db.Text
  status      project_status @default(draft)
  created_at  DateTime @default(now())
  updated_at  DateTime @updatedAt

  user        users            @relation(fields: [user_id], references: [id], onDelete: Cascade)
  versions    project_versions[]
  files       project_files[]
  messages    chat_messages[]

  @@index([user_id])
}

enum project_status {
  draft
  generated
  refined
}
```

### project_versions

```prisma
model project_versions {
  id             UUID   @id @default(uuid())
  project_id     UUID
  version_number Int
  label          String?
  created_at     DateTime @default(now())

  project        projects        @relation(fields: [project_id], references: [id], onDelete: Cascade)
  files          project_files[]
  messages       chat_messages[]

  @@unique([project_id, version_number])
  @@index([project_id])
}
```

### project_files

```prisma
model project_files {
  id          UUID   @id @default(uuid())
  project_id  UUID
  version_id  UUID
  path        String
  content     String @db.Text
  file_source file_source @default(ai_generated)
  updated_at  DateTime @updatedAt

  project     projects          @relation(fields: [project_id], references: [id], onDelete: Cascade)
  version     project_versions  @relation(fields: [version_id], references: [id], onDelete: Cascade)

  @@unique([version_id, path])
  @@index([project_id])
  @@index([version_id])
}

enum file_source {
  template
  ai_generated
  modified
}
```

### chat_messages

```prisma
model chat_messages {
  id          UUID   @id @default(uuid())
  project_id  UUID
  version_id  UUID
  role        message_role
  content     String @db.Text
  file_changes Json?
  created_at  DateTime @default(now())

  project     projects          @relation(fields: [project_id], references: [id], onDelete: Cascade)
  version     project_versions  @relation(fields: [version_id], references: [id], onDelete: Cascade)

  @@index([project_id])
  @@index([version_id])
}

enum message_role {
  user
  assistant
}
```

## Risks / Trade-offs

### Risk: File Content Size Limits

**Risk:** Storing large files as TEXT could impact query performance and database size.

**Mitigation:**
- Add validation at API level for reasonable file size limits
- Monitor database size in production
- Future: migrate large files to S3 with path reference

### Risk: JSONB Query Complexity

**Risk:** `file_changes` as JSONB may be difficult to query for analytics.

**Mitigation:**
- For MVP, only need to store and retrieve, not complex querying
- Future: add GIN indexes if querying becomes necessary

### Trade-off: No Soft Deletes

**Decision:** Using hard deletes with cascade behavior.

**Implication:** Deleted data cannot be recovered without database backups.

**Future consideration:** Add `deleted_at` timestamps if audit trail becomes requirement.

## Migration Plan

1. **Start Database:**
   ```bash
   docker-compose -f docker-compose.dev.yaml up -d
   ```

2. **Development (from project root):**
   ```bash
   cd apps/api
   pnpm prisma migrate dev --name init_database_layer
   pnpm prisma generate
   ```

3. **Staging/Production:**
   ```bash
   pnpm prisma migrate deploy
   ```

4. **Rollback Strategy:**
   - Keep migration files in git
   - Can rollback with `prisma migrate resolve`
   - Database backups required before deployment
